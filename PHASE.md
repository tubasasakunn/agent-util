# 実装フェーズ

## Phase 1: LLMクライアント

**目的:** Gemma APIと通信できるGoの基盤を作る

**概要:**
OpenAI互換APIを叩くHTTPクライアントを実装する。
リクエスト構築（messages, tools, response_format）、レスポンスパース、SLM出力のJSON補正ミドルウェアを含む。

**ゴール:**
- Goバイナリから Gemma に質問して回答を得られる
- JSON mode で構造化出力を取得できる
- SLMの壊れたJSON（`{null}` 等）を補正して安全にパースできる

---

## Phase 2: エージェントループ

**目的:** while(true)でモデルと対話を回し続けるループを作る

**概要:**
ユーザー入力→モデル呼び出し→応答表示→次の入力、の基本ループを実装する。
Continue/Terminalの状態遷移型でループの継続・終了を管理する。
この段階ではツール実行はなく、純粋な対話ループのみ。

**ゴール:**
- CLIでモデルとマルチターン対話ができる
- maxTurnsで安全に停止できる
- 状態遷移が型で管理されている

---

## Phase 3: ツール実行

**目的:** ADR-002のルーター＋単一ツールパターンを実装し、ツールを使えるエージェントにする

**概要:**
Tool interfaceを定義し、エージェントループにルーターステップ（JSON modeでツール選択）とサブエージェントステップ（単一ツールでFunction Calling）を組み込む。
ビルトインツール（shell, read_file等）で動作確認する。

**ゴール:**
- 「ファイルを読んで」→ルーターがread_fileを選択→実行→結果を自然言語で回答できる
- Tool interfaceに準拠すれば新しいツールを簡単に追加できる
- ツール結果のフィードバック→次の判断のマルチターンが動く

---

## Phase 4: コンテキスト基盤

**目的:** メッセージ履歴を管理し、コンテキスト使用量を把握できるようにする

**概要:**
メッセージ履歴の追加・参照・削除を管理する構造体を実装する。
トークン数の推定（文字数ベースまたはtokenizer）と使用率監視を行い、後続のPhaseで縮約・分割を判断するための基盤とする。

**ゴール:**
- メッセージ履歴がデータ構造として管理されている
- 現在のコンテキスト使用率（トークン数/上限）を常に把握できる
- 閾値超過時にイベントを発火できる（縮約トリガーの土台）

---

## Phase 5: コンテキスト縮約

**目的:** 8Kコンテキストでも長い対話を維持できるようにする

**概要:**
コンテキストが溢れそうになった時の多段階縮約カスケードを実装する。
ツール結果のバジェット制限（文字数上限、超過分のファイル退避）→観測マスキング（古いツール出力を隠蔽、呼び出し自体は可視に保持）→古いメッセージのSnip（粗い切り捨て）→LLMによるCompact（要約圧縮）の順に試す。

**設計根拠（記事より）:**
- コンテキストロット: キーコンテンツがウィンドウ中央で30%+の性能低下（Stanford "Lost in the Middle"）
- 観測マスキング: JetBrains Junieパターン。ツール出力は最も大きくかつ最も早く陳腐化する
- ACON研究: 推論トレース > 生ツール出力を優先することで、26〜54%のトークン削減で精度95%+維持

**ゴール:**
- ツール結果が大きくてもバジェット制限で溢れない
- 観測マスキングにより古いツール出力を効率的に除去できる
- 長い対話でも古いメッセージが自動的に整理される
- 縮約しても直近のコンテキスト（直近N件）は必ず保持される

---

## Phase 6: コンテキスト分割とサブエージェント

**目的:** 単一コンテキストに収まらないタスクを分割して処理できるようにする

**概要:**
サブエージェント的なコンテキスト分離の仕組みを実装する。
タスクを独立したコンテキストで実行し、結果だけを親に返す構造（1,000〜2,000トークンの凝縮サマリー）。
3つの実行モデル（Fork/Worktree/Coordinator）とRalph Wiggumループの2フェーズパターンを含む。

**サブエージェントの3実行モデル:**
- **Fork** — 親コンテキストのコピーで独立実行（並列探索向け）
- **Worktree** — ファイルシステムを物理分離（git worktree）。変更の衝突を防止
- **Coordinator** — メールボックスシステムでタスク分配と結果統合

**Ralph Wiggumループ: 2フェーズパターン（記事より）:**
1. **Initializerエージェント** — 環境セットアップ（init script、progressファイル、機能リスト、初回git commit）
2. **Codingエージェント** — 毎セッションでgitログとprogressファイルから状況把握→最優先の未完了機能を選択→実装→commit→サマリー記録

**設計根拠:** ファイルシステムがコンテキストウィンドウを超えた継続性を提供する。

**ゴール:**
- 大きなタスクを小さなサブタスクに分割して別コンテキストで実行できる
- 各サブタスクの結果を統合して最終応答を生成できる
- コンテキスト上限に達しても状態を保存して継続できる
- 親エージェントの中断時にサブエージェントを明示的に終了制御できる

---

## Phase 7: コンテキスト構成

**目的:** コンテキストの中身の並べ方・注入方法を最適化する

**概要:**
システムプロンプトの静的/動的分離、必要な知識のオンデマンド注入（累進的開示）を実装する。
ツール定義の動的フィルタリング（ルーターが既に担っている部分の拡張）、MEMORYインデックスパターン（ポインタだけ常時載せ、実体はオンデマンド読み込み）を含む。

**プロンプト構築の優先度スタック（記事より）:**
```
1. システムメッセージ（最高優先）
2. ツール定義
3. 開発者指示（設定ファイル等）
4. ユーザー指示
5. 会話履歴（最低優先）
```

**配置の原則:**
- 重要情報はプロンプトの先頭と末尾に配置（"Lost in the Middle"対策）
- 中央に配置された情報は性能を30%+低下させる
- ツールスコーピング: 現在のステップに必要な最小限のツールのみ公開（Vercelはツール80%削除で性能向上）

**ゴール:**
- システムプロンプトが静的部分と動的部分に分離されている
- 必要な知識だけがオンデマンドでコンテキストに注入される
- 重要情報がウィンドウの先頭/末尾に配置されている
- 8Kの限られたコンテキストで情報密度が最大化されている

---

## Phase 8: 検証ループとエラー回復

**目的:** エラーの複合を抑制し、出力品質を安定させる

**概要:**
エージェントのステップごとにエラー分類と回復戦略を適用する。
さらに、ツール実行後の検証ループ（ルールベース検証、LLM-as-judge）を組み込み、
PEV（Plan-Execute-Verify）サイクルをエージェントループの正式な構造として実装する。

**設計根拠（記事より）:**
- エラー複合: 10ステップ × 95%/ステップ = 59.9% E2E成功率（SLLMの現実）
- 検証手段の提供で品質2〜3倍向上（Boris Cherny / Claude Code）
- Stripe本番ではリトライ上限を2回に制限

**エラーの4分類（LangGraph）:**
1. **一時的（transient）** — 指数バックオフでリトライ（最大32秒、25%ジッタ）
2. **LLM回復可能** — エラーをツール結果として返却、モデルが自己修正
3. **ユーザー修正可能** — 人間の介入を要求（interrupt）
4. **予期しない** — デバッグ用にバブルアップ、安全停止

**検証の3アプローチ:**
1. **ルールベース** — テスト、リンター、型チェッカー（決定論的、低コスト）
2. **ビジュアル** — スクリーンショット比較（UI向け、将来拡張）
3. **LLM-as-judge** — 独立サブエージェントが出力を評価（意味論的、高コスト）

**ゴール:**
- エラーが4分類に基づいて適切に処理される（リトライ/自己修正/人間委譲/停止）
- 連続失敗にキャップが設定されている（MAX_CONSECUTIVE_FAILURES）
- ツール実行後にルールベース検証が走り、失敗時に自動修正ループに入る
- PEVサイクルが型として表現され、Verifyなしに完了判定しない

---

## Phase 9: 権限とガードレール

**目的:** エージェントの行動を安全に制約する多層防御を実装する

**概要:**
モデルの推論から権限強制をアーキテクチャ的に分離する。
入力/出力/ツール呼び出しの3層ガードレールとトリップワイヤ機構、
deny→ask→allowのパーミッションパイプラインを実装する。

**設計根拠（記事より）:**
- Anthropic: モデルが「何を試みるか」を決定、ツールシステムが「何を許可するか」を決定
- OpenAI SDK: 入力/出力/ツール呼び出しの3層 + トリップワイヤ（即時停止）
- Claude Code: ~40のツール能力を独立してゲーティング

**3層ガードレール:**
1. **入力ガードレール** — ユーザー入力の検証（インジェクション検知等）
2. **ツール呼び出しガードレール** — 全ツール呼び出し時に権限チェック
3. **出力ガードレール** — 最終出力の安全性検証

**パーミッションパイプライン:**
```
ツール実行要求
  → Denyリスト照合（静的ルール）→ 早期拒否
  → Allowリスト照合 → 即許可
  → ツール固有チェック → 動的判定
  → ユーザー承認（最終ゲート）
  → 監査ログ（記録）
```

**ゴール:**
- ツール実行前にdeny→ask→allowパイプラインが必ず通過する
- デフォルトがfail-closed（明示許可なしは拒否）
- トリップワイヤ発動でエージェントループが即時停止する
- ReadOnlyツールは自動承認、変更ツールはユーザー確認を要求
- 権限チェックの結果が監査ログとして記録される

---

## Phase 10: JSON-RPCサーバー

**目的:** ADR-001のJSON-RPC over stdioを実装し、多言語ラッパーからGoコアを操作可能にする

**概要:**
stdin/stdoutでJSON-RPCメッセージを送受信するサーバーを実装する。
ツール登録（ラッパー→コア）、ツール実行要求（コア→ラッパー）の双方向通信を実現する。
Python/JSラッパーからGoバイナリをsubprocess起動して利用する形態。

**ゴール:**
- Python/JSからGoバイナリをsubprocess起動し、JSON-RPCでツール登録→タスク実行ができる
- ツールの実装はラッパー側で、Goコアは呼び出し判断だけ担う
- ストリーミングでリアルタイムに進捗通知ができる
