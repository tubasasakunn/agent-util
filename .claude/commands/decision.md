# Decision Log

引数: $ARGUMENTS

プロジェクトの技術的な判断をADR（Architecture Decision Record）として記録・参照する。
記録は `.claude/decisions/` に保存する。

## ワークフロー

引数に応じて3つのモードで動作する。

### モード1: 記録（引数なし）

会話の文脈から技術的な判断を抽出して記録する。

1. `.claude/decisions/` の既存ファイルを確認し、次の連番を決定する
2. 会話から以下を抽出する:
   - 何を判断したか
   - どんな選択肢を検討したか
   - 何を選び、なぜ選んだか
   - この判断による影響
3. 記録テンプレートに従ってドラフトを作成し、ユーザーに確認する
4. 確認後、`.claude/decisions/NNN-<slug>.md` として保存する
5. 既存の判断と矛盾する場合、古い方の `status` を `superseded` に更新する

### モード2: 一覧（引数が `list`）

`.claude/decisions/` 内の全ファイルを読み、以下の形式で一覧表示する:

```
001 - JSON-RPC over stdioを採用 (2026-04-18) [accepted]
002 - Goをコア言語に採用 (2026-04-19) [accepted]
```

### モード3: 検索（その他の引数）

`.claude/decisions/` 内のファイルをGrepで検索し、引数に関連する判断を見つけて内容を表示する。
複数マッチした場合は全て表示する。該当なしの場合はその旨を伝える。

このモードはClaude自身が新たな技術判断をする際にも活用する。過去の判断との一貫性を保つため、関連するキーワードで検索してから新しい判断を提案すること。

## 記録テンプレート

ファイル名: `NNN-<slug>.md`（NNNは連番3桁、slugは英語kebab-case）

```markdown
---
id: "NNN"
title: 判断のタイトル
date: YYYY-MM-DD
status: accepted
---

## コンテキスト
なぜこの判断が必要になったか。背景と制約。

## 検討した選択肢
各選択肢の概要とメリット・デメリット。

## 判断
何を選んだか。一文で。

## 理由
なぜその選択肢を選んだか。他を退けた理由も含める。

## 影響
この判断によって何が決まり、何が制約されるか。
```

statusの値: `accepted`（有効）、`superseded`（後続の判断で置き換え済み）、`deprecated`（非推奨）
superseded の場合は `superseded_by: NNN` を追加する。

## 注意事項

- 記録する前に必ずユーザーに内容を確認する
- 判断は日本語で記述する（技術用語は原語のまま）
- 1つの判断につき1ファイル。複数の判断を1ファイルにまとめない
