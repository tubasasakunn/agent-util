# メモリアーキテクチャとコンテキスト管理

## 3層メモリアーキテクチャ

情報の鮮度と抽象度に応じて記憶を分類する。

| 層 | 内容 | 目的 | 実装例 |
|---|---|---|---|
| L1: エピソディック | 生のログ、個別の行動・推論・結果 | 過去の特定の瞬間を正確に再現 | JSONL、日次ログ |
| L2: セマンティック | 統計的傾向、因果関係、パターン | 生データから洞察を抽出・検索可能に | 全文検索、ベクトル埋め込み |
| L3: 戦略的 | 禁止事項、行動指針、暗黙知 | 過去の経験をリアルタイムの判断基準へ変換 | 設定ファイル、プレフィックス注入 |

## メモリ昇格プロセス

L1→L2→L3への「昇格」がアーキテクチャの核:

```
L1（生ログ）
  ↓ リフレクション・エンジンが分析
L2（パターン抽出）例: 「この種のエラーは73%の確率でX原因」
  ↓ 強力なパターンが昇格
L3（アクティブ戦略）例: 「このエラーを見たらまずXを確認する」
```

エージェントは単に過去を思い出すだけでなく、経験から「感覚」を養うことが可能になる。

## 時間的減衰（Temporal Decay）

エビングハウスの忘却曲線を模したメモリライフサイクル:

```
Active → Warm → Archive → Forget
```

- 重要度の低いメモリは時間とともにフェードアウト
- 頻繁に参照される情報はセマンティック記憶へ固定
- エッジデバイスのストレージ制約下でリソース最適化

## コンテキスト縮約カスケード

SLLMのコンテキスト制約（8Kトークン等）に対する多段階戦略:

```
段階1: ツール結果のバジェット制限
  ├ 各ツールの最大出力文字数を制限
  └ 超過分はファイルに退避

段階2: 古いメッセージのSnip（粗い切り捨て）

段階3: Microcompact（古いツール結果をLLMで要約）
  └ 直近N件は必ず保持

段階4: Autocompact（閾値超過で発動）
  └ サブエージェントが構造化要約を生成

段階5: Reactive Compact（エラー時の最終手段）
  └ 強制的に圧縮して再試行
```

**SLLMでの重要性:** 8Kコンテキストでは、ツール結果1回で容量の大部分を消費しうる。
カスケードの設計が品質を直接左右する。

## MEMORY.mdインデックスパターン

常時コンテキストに載せるのは軽量ポインタだけ:

```markdown
# MEMORY.md（常時ロード、~150文字/行）
- [JSON-RPC採用](decisions/001-json-rpc.md) — 通信方式の判断
- [Goコア設計](decisions/002-go-core.md) — 言語選定
```

実体は個別ファイルに分散配置。必要時だけ読み込む。

**Strict Write Discipline:**
- 書き込み成功を確認してからインデックス更新
- 保存された知識は「権威的事実」ではなく「ヒント」
- バージョン番号と信頼度スコアを付与

## Sakana AI NAMMs: 適応的忘却

Neural Attention Memory Models — 進化アルゴリズムで忘却戦略を最適化。

- どのアテンション情報を残し、何を捨てるかを最適化
- 勾配降下法では学習できない二値的決定を進化的に解決
- 特定ドメインで進化した戦略が他ドメインにも転移可能

**このプロジェクトへの示唆:** コンテキスト管理の自動化。
「何を残すか」のポリシーをハードコードするのではなく、タスク成功率をフィードバックとして適応的に調整する機構を検討。

## プロンプトキャッシュ最適化

```
[静的部分] ← キャッシュ可能
  システムプロンプト本体
  ツール定義
  基本設定（CLAUDE.md等）
[__DYNAMIC_BOUNDARY__]
[動的部分] ← キャッシュ不可
  ユーザー発話
  ツール実行結果
  セッション状態
```

キャッシュは先頭一致で適用されるため、静的部分を前、動的部分を後ろに配置。
**キャッシュブレークベクター**（キャッシュを無効化する要因）を追跡:
- タイムスタンプの混入
- ユーザー名の差し込み
- 乱数シードの注入
