# ハーネスパターン

Claude Codeの流出コード分析(512,000行TypeScript)とLangChain等の知見から抽出された設計パターン。

## パターン1: ハーネスパラダイム — モデルと制御の分離

ハーネスが解決する5つの課題:
1. アクション空間の制約（ツールで行動範囲を限定）
2. 会話状態の管理
3. パーミッションによる安全性確保
4. 障害の適切なハンドリング
5. リソース使用量の最適化

**いずれもモデルの賢さでは解決できない。** モデルは統計的にもっともらしい応答を返す装置であり、ハーネスは「もっともらしいが危険/非効率な応答」を実行前に弾くフィルタ。

## パターン2: ツールコントラクト — 統一インターフェース

全ツールが統一された契約を実装する。Claude Codeでは:

```
Tool = {
  name: string
  inputSchema: ZodSchema        // 入力の型定義
  behaviors: {
    isConcurrencySafe: bool     // 並列実行可能か（デフォルトfalse）
    isReadOnly: bool            // 読み取り専用か
    requiresPermission: bool    // 権限チェック必要か
  }
  permissionCheck: fn           // 権限チェック
  execute: fn                   // 実行ロジック
}
```

**設計意図:** ハーネスがツールの内部実装を理解せずとも、振る舞いについて推論できる。
`isConcurrencySafe=false`のツールは並列実行されず、`isReadOnly=true`は自動承認。

## パターン3: クエリエンジン — 中央オーケストレータ

会話ライフサイクルの中央制御。Continue Sitesパターンで段階的な回復戦略:

```
エラー発生
  → コンテキスト折りたたみ（古い情報の要約）
  → リアクティブコンパクション（急激なトークン消費への対応）
  → トークンエスカレーション（より大きなコンテキスト枠へ）
  → モデルエスカレーション（上位モデルへ切り替え）
  → 安全停止（繰り返し失敗時の強制中断）
```

**実運用の知見:** Claude Codeでは連続50回失敗で25万APIコール/日が浪費された事例から、MAX_CONSECUTIVE_FAILURES=3に制限。

## パターン4: パーミッションシステム — 多層防御

パイプライン型の多層評価:

```
1. Deny/Allowリスト（静的ルール）→ 早期拒否
2. ツール固有チェック（Bashなら23項目）
3. 分類器による動的判定（インジェクション検知等）
4. ユーザー承認（最終ゲート）
5. 監査ログ（記録）
```

**原則: 早期拒否・晩期承認。** 明らかに危険なものを最初に捨て、判断が難しい中間帯だけを人間やLLMに回す。

## パターン5: コンテキストエントロピー管理

MEMORY.mdを「データではなくポインタのインデックス」として扱う。
常時コンテキストに載るのは150文字のポインタだけ、実体は個別ファイル。

**Strict Write Discipline:** 保存された知識を「権威的事実」ではなく「ヒント」として扱う。
LLMが生成した要約を事実として固定するとハルシネーションが永続化するリスクがある。

## パターン6: プロンプトキャッシュ最適化

```
[静的部分] ← セッション間でキャッシュ共有可能
  システムルール、ツール定義、基本設定
[__DYNAMIC_BOUNDARY__]
[動的部分] ← セッション固有
  ユーザー発話、ツール実行結果、セッション状態
```

静的部分を前、動的部分を後ろに配置。キャッシュは先頭から一致する範囲に適用されるため。

## パターン7: フラストレーション検出

正規表現でユーザーの感情状態を推論。LLM呼び出しより圧倒的に低コスト。
**原則: 安い・速い・決定論的な処理で済むものはLLMを使わない。**

## パターン8: PEVサイクル（Plan-Execute-Verify）

```
1. Plan  — タスク分解 + 「どうなれば成功か」の検証基準を定義
2. Execute — ファイル操作、コード実行
3. Verify — テスト実行結果という「外部の真実」でフィードバック
```

モデル自身の判断ではなく、テスト結果をフィードバックの核とする。

## パターン9: Ralph Wiggumループ — 永続化

エージェントがコンテキスト制限に達した場合:
1. ハーネスが「終了」を監視
2. 目的未達成ならコンテキストをリセット
3. ファイルシステムに書かれた「前回の状態」を読み込ませて再開

単一の会話窓に縛られない超長時間稼働を実現。

## パターン10: セマンティック・ハンドシェイク

マルチエージェントや外部APIとの統合時:
1. 通信開始時にサポートするコンテキストのスキーマを確認し合う
2. 曖昧なパラメータ（日付形式、通貨単位等）がある場合は実行前にQuery for Clarification
3. ツール呼び出しの失敗率を劇的に下げ、意図しない破壊的操作を防止

## 3層制御レイヤーの設計順序

| 優先順位 | レイヤー | 目的 |
|---|---|---|
| 1位 | 制約ハーネス | 生成前に解決空間を縮小。失敗の母数を最小化 |
| 2位 | フィードバックループ | 出力後の自己修正。人間を介さず品質向上 |
| 3位 | 品質ゲート | 最終的な決定論的基準。前2層を潜り抜けた問題を阻止 |

不変条件（invariants）を強制し、その中でエージェントに自由度を持たせる。
