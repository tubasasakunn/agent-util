# 003: Phase 5 コンテキスト縮約カスケード検証

## 目的

Phase 5で実装した4段階縮約カスケード（BudgetTrim → ObservationMask → Snip → Compact）が、実際のエージェント利用シナリオで正しく動作するか確認する。ユニットテスト（internal/context/compaction_test.go）とは異なり、エンドツーエンドのシナリオで検証する。

### 検証項目

| # | テスト | 確認すること |
|---|---|---|
| 1 | BudgetTrim大ファイル | 5000文字のツール結果がBudgetTrimで切り詰められ、エージェントループが正常完了する |
| 2 | 多ターン溢れ防止 | 500トークンコンテキストで10ターン対話しても溢れない |
| 3 | ToolCallペア整合性 | 5回のツール実行後に縮約してもToolCall-ToolResultペアが分断されない |
| 4 | 観測マスキング推論保持 | 古いツール出力がマスクされても推論トレース（ToolCalls）が保持される |
| 5 | 8Kリアルシミュレーション | 8192トークンで20ターン（うち10回ツール実行）の現実的シナリオ |

## 手段

- 方法: Go テストプログラム（mockCompleter使用、SLLM不要）
- 対象: `internal/context.Manager` + `internal/engine.Engine`
- 実行: `go test -v ./.claude/skills/investigation/003_phase5_context_compaction/`

## 結果

### 総合評価

| # | テスト | 結果 | 判定 |
|---|---|---|---|
| 1 | BudgetTrim大ファイル | 5000文字→500文字に切り詰め、2ターンで正常完了 | **PASS** |
| 2 | 多ターン溢れ防止 | Turn 7で縮約発動（400→114 tok）、最終214/500 tok (43%) | **PASS** |

### 詳細

#### シナリオ 1: BudgetTrim 大ファイル
- 5000文字のツール結果（read_file模擬）を BudgetMaxChars=500 で切り詰め
- エージェントループ（ルーター→ツール実行→最終応答）が2ターンで正常完了
- 切り詰め後もルーターが正しく "none" を選択し、最終応答を生成

#### シナリオ 2: 多ターン溢れ防止（500トークン）
- 500トークンの小さなコンテキストで10ターン対話
- Turn 7 で閾値80%に到達、縮約発動
- 400→114トークンに縮約（BudgetTrim→ObservationMask→Snip の3段階）
- 最終状態: 9メッセージ、214/500トークン (43%)、直近メッセージ保持

#### シナリオ 3: ToolCallペア整合性
- 5回のツール実行サイクル（各: user + assistant/tool_calls + tool）= 15メッセージ
- 縮約後: 7メッセージ（マーカー1 + 直近2サイクル6件）
- ToolCall 2ペア残存、孤児なし

#### シナリオ 4: 観測マスキングの推論トレース保持
- 古いtool結果（50行、1050文字）が `[observation masked]` に置換（20文字）
- 83%のトークン削減
- assistant の ToolCalls（ツール名 + 引数）は完全に保持
- 新しいtool結果は変更なし

#### シナリオ 5: 8K リアルシミュレーション
- 8192トークン、reserved=58（システムプロンプト+ツール定義）
- 20ターン実行（うち偶数ターンでツール実行、ファイルサイズ500〜3350文字）
- Turn 15 で閾値80%到達、縮約1回発動
- 縮約前: 48メッセージ → 縮約後: 3312/8192 tok (40%)
- 最終状態: 60メッセージ、5968/8192 tok (73%)、10ToolCallペア全整合

### 設計への示唆

1. **カスケードの有効性確認**: BudgetTrim→ObservationMask→Snip の3段階で、Compact（LLM要約）なしでも20ターンの対話を維持できた
2. **KeepLast=6 の妥当性**: 直近2サイクル分の保護で、推論の連続性が維持された
3. **TargetRatio=0.6 の余裕**: 縮約後40%まで下がり、次の閾値到達まで5ターン以上の余裕があった
4. **Compact（LLM要約）は当面不要**: Stage 1-3 だけで十分な縮約効果。SLLMでの要約品質検証は後続フェーズで行う
5. **ToolCallペア整合性**: adjustKeepBoundary() が正しく動作し、全シナリオで孤児ゼロ
