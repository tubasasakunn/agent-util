# エージェント設計ナレッジベース

このプロジェクト（SLLM向けエージェントハーネス）の設計判断を支える基礎知識。
ハーネスエンジニアリング、Claude Codeの設計分析、Sakana AIの進化的アプローチ等から抽出。

## 中核原則

**「エージェント = モデル + ハーネス」** — モデルは交換可能、ハーネスが本体。

| 原則 | 意味 |
|---|---|
| モデルと制御の分離 | モデルの賢さで解決できない問題（権限、リソース制限、安全性）はハーネスが担う |
| fail-closed | 明示的に許可されない限り、安全側に倒す |
| 決定論的な制約 > 確率的な準拠 | 「従え」とプロンプトで指示するより、リンターでブロックする |
| 累進的開示 | 全情報を一度に渡さず、必要時に必要な情報だけ提供 |
| 外部の真実でフィードバック | モデル自身の判断ではなく、テスト結果やログという外部事実を核とする |

## ハーネスの3層制御

```
1. 制約ハーネス — 生成前に解決空間を縮小（ルールファイル、型システム）
2. フィードバックループ — 出力後の自己修正（Write-Test-Fix）
3. 品質ゲート — 最終的な決定論的チェック（CI/CD、人間の承認）
```

## エージェントの4コアコンポーネント（Shachi設計）

Sakana AIのShachiフレームワークが提唱する、エージェントの標準アーキテクチャ:

| コンポーネント | 役割 | このプロジェクトでの対応 |
|---|---|---|
| **LLM** | 思考と意思決定 | OpenAI互換API経由の推論モジュール |
| **Tools** | 環境との相互作用 | JSON-RPCで呼び出されるツール群 |
| **Memory** | 過去の記録と検索 | 3層メモリ（エピソディック→セマンティック→戦略的） |
| **Config** | 静的なアイデンティティ | システムプロンプト、制約、設定ファイル |

## このプロジェクトへの設計指針

このプロジェクトはGoバイナリ + JSON-RPC over stdioで多言語ラッパーから呼ばれるエージェントハーネス。
以下をGoコアの設計に組み込むこと:

1. **ツール契約の統一** — 全ツールが同一interface（Name, Description, Parameters, Execute）を実装
2. **fail-closed権限** — deny→ask→allowの3値、デフォルトは安全側
3. **コンテキスト縮約カスケード** — SLLMのコンテキスト制約に対する多段階の回復戦略
4. **パース補正** — SLMの不完全なJSON出力を補正するミドルウェア
5. **PEVサイクル** — Plan→Execute→Verifyのループを基本構造に
