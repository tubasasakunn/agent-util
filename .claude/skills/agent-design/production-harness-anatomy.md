# プロダクションハーネスの解剖学

Anthropic, OpenAI, LangChain等の実装分析から抽出された、プロダクション品質ハーネスの体系的知見。
出典: "The Anatomy of an Agent Harness" (2026年)

## スキャフォールディング・メタファー

Beren Millidge (2023) の「Scaffolded LLMs as Natural Language Computers」による精密な比喩:

```
生のLLM = RAM・ディスク・I/Oのないプロセッサ

コンテキストウィンドウ → RAM（高速だが有限）
外部データベース     → ディスク（大容量だが低速）
ツール統合           → デバイスドライバ
ハーネス             → オペレーティングシステム
```

**スキャフォールディング原則:** 建設の足場と同様、ハーネスはモデルが単独では到達できない能力を実現する一時的インフラ。モデルが改善すればハーネスの複雑さは減少すべき。

## エンジニアリングの3層

```
┌─────────────────────────────────────┐
│  ハーネスエンジニアリング            │ ← アプリケーション全体
│  ┌─────────────────────────────┐    │
│  │  コンテキストエンジニアリング │    │ ← 何をいつ見せるか
│  │  ┌─────────────────────┐    │    │
│  │  │ プロンプトエンジニアリング│    │    │ ← モデルへの指示
│  │  └─────────────────────┘    │    │
│  └─────────────────────────────┘    │
└─────────────────────────────────────┘
```

ハーネスエンジニアリングはプロンプトとコンテキストの両方を包含し、さらにツールオーケストレーション、状態永続化、エラー回復、検証ループ、安全性強制、ライフサイクル管理を加えた完全なシステム。

### プロンプト構築の階層（#5）

```
優先度高 → 低:
1. サーバー制御システムメッセージ（最高優先）
2. ツール定義
3. 開発者指示
4. ユーザー指示（AGENTS.md等、32KiB制限）
5. 会話履歴
```

重要情報はプロンプトの先頭と末尾に配置（"Lost in the Middle"対策）。

### 検証ループ（#10）

Boris Cherny（Claude Code作者）: 検証手段を与えると品質が2〜3倍向上。

```
3つの検証アプローチ:
1. ルールベース — テスト、リンター、型チェッカー（決定論的）
2. ビジュアル  — Playwright等でスクリーンショット検証（UI向け）
3. LLM-as-judge — 独立サブエージェントが出力を評価（意味論的）
```

Martin Fowlerのフレーム:
- **ガイド（feedforward）** — 行動前に軌道修正
- **センサー（feedback）** — 行動後に観測・修正

### エラー複合の定量化（#8）

```
10ステップ × 99%/ステップ = 90.4% E2E成功率
10ステップ × 95%/ステップ = 59.9% E2E成功率  ← SLMの現実
```

LangGraphの4分類:
1. **一時的** — バックオフでリトライ
2. **LLM回復可能** — エラーをToolMessageとして返却、モデルが調整
3. **ユーザー修正可能** — 人間の介入を要求
4. **予期しない** — デバッグ用にバブルアップ

Stripe本番: リトライ上限を2回に制限。

## ループの動作: 7ステップ

```
Step 1: プロンプトアセンブリ
  system + tools + memory + history + user message
    ↓
Step 2: LLM推論
  テキスト、tool_calls、またはその両方を生成
    ↓
Step 3: 出力分類
  tool_callsなし → 最終回答（ループ終了）
  tool_callsあり → Step 4へ
  handoff要求   → エージェント切替、Step 1へ
    ↓
Step 4: ツール実行
  引数検証 → 権限チェック → サンドボックス実行 → 結果キャプチャ
  読み取り専用: 並列実行可  |  変更あり: 直列実行
    ↓
Step 5: 結果パッケージング
  LLM可読形式にフォーマット。エラーもエラー結果として返却
    ↓
Step 6: コンテキスト更新
  結果を会話履歴に追加。閾値超過時は縮約を発動
    ↓
Step 7: ループ
  → Step 1へ戻る
```

**終了条件（6層）:**
1. tool_callsなしの応答
2. 最大ターン数超過
3. トークンバジェット枯渇
4. ガードレールトリップワイヤ発動
5. ユーザー中断
6. 安全性拒否

## フレームワーク実装比較

### Anthropic Claude Agent SDK

```
query() → async iterator（ストリーミング）
```

- ランタイムは「ダムループ」。全知能はモデルに存在
- **Gather-Act-Verify サイクル:** コンテキスト収集 → 行動 → 検証 → 反復
- メモリ: CLAUDE.md + auto-generated MEMORY.md（3層: インデックス → トピックファイル → トランスクリプト検索）
- メモリは「ヒント」扱い。実行前に実状態を検証

### OpenAI Agents SDK

```
Runner: async / sync / streamed の3モード
```

- コードファースト。ワークフローをPythonで直接記述（グラフDSLなし）
- ツール3種: function tools（@function_tool）、hosted tools（WebSearch, CodeInterpreter, FileSearch）、MCP
- エージェント間: agents-as-tools（スペシャリストが限定サブタスク） / handoffs（制御の完全移譲）
- ガードレール: 入力 / 出力 / ツール呼び出しの3層 + トリップワイヤ
- **Codex 3層アーキテクチャ:** Core → App Server（双方向JSON-RPC） → クライアント面（CLI, VS Code, Web）

### LangGraph / LangChain

```
明示的な状態グラフ: llm_call ノード ←→ tool_node ノード → END
```

- 状態を型付き辞書として管理、リデューサーで更新をマージ
- スーパーステップ境界でチェックポイント（中断後再開、タイムトラベルデバッグ）
- AgentExecutor（v0.2で非推奨）→ LangGraphへ進化（拡張性・マルチエージェント対応）
- **Deep Agents:** 組み込みツール + write_todos計画 + ファイルシステムコンテキスト + サブエージェント + 永続メモリ

### CrewAI

```
Agent（ハーネス）→ Task（作業単位）→ Crew（エージェント集合）
```

- ロールベースのマルチエージェント（role, goal, backstory, tools）
- **Flows層:** 決定論的バックボーン + 知能が必要な部分だけCrew
- ルーティングと検証はFlows、自律的協調はCrews

### AutoGen (→ Microsoft Agent Framework)

```
3層: Core → AgentChat → Extensions
```

- 会話駆動型オーケストレーション
- 5パターン: sequential, concurrent（fan-out/fan-in）, group chat, handoff, magentic
- **Magentic:** マネージャーエージェントが動的タスク台帳を管理、スペシャリストを調整

## 7つのアーキテクチャ判断

全ハーネス設計者が直面する選択:

### 1. シングル vs マルチエージェント

**原則: まずシングルを最大化。** AnthropicもOpenAIも同意。
分割の基準: ツール過多（10+の重複ツール）、明確に分離されたタスクドメイン。
マルチの代償: ルーティング用LLM呼び出し増加、ハンドオフ時のコンテキスト喪失。

### 2. ReAct vs Plan-and-Execute

| 方式 | 特徴 | トレードオフ |
|---|---|---|
| ReAct | 毎ステップで推論と行動を交互 | 柔軟だがステップ単価が高い |
| Plan-and-Execute | 計画と実行を分離 | LLMCompilerで3.6倍高速化の報告 |

### 3. コンテキストウィンドウ管理戦略

5つのプロダクション手法:
1. **時間ベースクリア** — 古いメッセージの単純削除
2. **会話要約** — LLMによる履歴圧縮
3. **観測マスキング** — 古いツール出力を隠蔽（呼び出し自体は可視）
4. **構造化ノートテイキング** — 進行中の要約を別管理
5. **サブエージェント委譲** — 広く探索、凝縮して返却

ACON研究: 26〜54%のトークン削減で精度95%+を維持（推論トレース>生ツール出力を優先）。

### 4. 検証ループ設計

- **計算的検証**（テスト、リンター）→ 決定論的な真実
- **推論的検証**（LLM-as-judge）→ 意味論的問題を捕捉、レイテンシ増

### 5. 権限・安全性アーキテクチャ

- **寛容型** — 高速だがリスク（大半を自動承認）
- **制限型** — 安全だが低速（承認を都度要求）

### 6. ツールスコーピング戦略

**ツールが多い = 性能低下。**
- Vercel: v0からツールの80%を削除 → 性能向上
- Claude Code: 遅延読み込みで95%のコンテキスト削減
- **原則: 現在のステップに必要な最小限のツールセットを公開**

→ ADR-002（ルーター+単一ツールパターン）はこの原則の実装。

### 7. ハーネスの厚さ

```
薄いハーネス（Anthropic志向）
  → モデル改善に賭ける。ダムループ + 賢いモデル
  → Anthropicは新モデル版でプランニング工程を定期削除

厚いハーネス（グラフベースFW志向）
  → 明示的制御に賭ける。フロー定義 + 状態グラフ
  → LangGraph、CrewAI Flows

SLLMの場合:
  → モデル能力が限定的なため、厚めのハーネスが必要
  → ただし「将来テスト」を意識: より強力なモデルで
    ハーネス複雑性を追加せずに性能向上するか？
```

## 共進化原則

モデルは特定のハーネスとともにpost-trainingされる。

- Claude Codeのモデルは、訓練時に使用したハーネスのツールに最適化されている
- ツール実装を変更すると、この密結合により性能が低下しうる
- **示唆:** SLLMをファインチューニングする場合、このハーネスのツール定義を訓練データに含める
