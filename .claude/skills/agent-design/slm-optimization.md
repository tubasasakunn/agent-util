# SLM（小規模言語モデル）最適化

SLLM向けエージェントハーネスの設計で特に重要な課題と解決策。

## 核心的課題: 構文と意味論の乖離

SLMはファインチューニングで特定形式の応答を「学習」するが、それは統計的な模倣に留まりがち。

**例:** 「ニューヨーク行きのチケットを予約して」
- SLMは構文的に完璧な予約コマンドを生成する
- しかし「JFK空港」か「LGA空港」かの曖昧さを解消せずに実行してしまう
- 形式（Syntax）には同意しているが、意味（Semantics）では合意に至っていない

**ハーネスの役割:** この乖離をモデルの外側で補正する。

## 解決策1: パース補正ミドルウェア

SLMの出力はJSONとして不完全なことが多い。ハーネス側で補正:

```
SLMの出力
  ↓
1. 末尾の不完全なJSON閉じ括弧を補完
2. シングルクォートをダブルクォートに変換
3. 末尾カンマの除去
4. tool_callsフィールドの欠損チェック
5. 引数の型変換（文字列→数値等）
  ↓
正規化されたツール呼び出し
```

**リトライ戦略:**
- パースエラー時: エラーメッセージをLLMに返して再生成を促す
- 3回失敗: フォールバック（より単純なプロンプトで再試行）
- SLMは修正指示を理解する能力が限定的なため、具体的な修正例を含める

## 解決策2: Thinkingトークン（テストタイム・スケーリング）

回答出力前にモデルが内部的な推論チェーンを展開するための計算領域。

| 手法 | 概要 | SLMへの効果 |
|---|---|---|
| SyncThink | 推論の飽和を検出し、無駄なトークン生成を抑制 | 推論品質を落とさずトークン節約 |
| AutoThink | 難易度に応じて思考トークン量を動的調整（難問90%、平易20%） | 限られたトークンバジェットの最適配分 |
| BATS | ツール呼び出しコストを認識し、「深い思考」vs「広い検索」を判断 | エージェントの行動計画最適化 |

**このプロジェクトへの示唆:**
Gemma 4 E2Bのようなモデルでは、Thinkingトークンの導入で推論品質が大幅に改善しうる。
ハーネス側でThinkingフェーズを明示的に制御する機構を検討。

## 解決策3: Shanksフレームワーク（Think-while-Listen）

音声/リアルタイム対話向けだが、概念はSLMエージェントにも適用可能:

```
従来: Listen → Think → Act（直列）
Shanks: Listen しながら Think、確信が得られた段階で Act（並列）
```

- 発話途中で必要なAPIコールの56.9%を完了
- ユーザーの誤りを即座に指摘（精度37%向上）

**エージェントへの応用:**
ユーザー入力の解析と並行して、予測されるツール呼び出しの準備を開始する。

## Google PLE: エッジでの推論効率

Per-Layer Embeddings — Gemma 4のE2B/E4Bで導入。

- 各デコーダー層に専用の調整経路を追加
- 初期埋め込みに全情報を詰め込む必要がなくなる
- わずかなパラメータ追加で各層の専門化を実現
- アクティブ計算量（FLOPs）を抑えつつ有効パラメータ数を増加

**意味:** 同じパラメータ数でもPLE搭載モデルは「効率的に賢い」。
Gemma 4 E2Bをエージェントのバックエンドにする場合、この特性を活かした設計が有効。

## セマンティック・ハンドシェイク・プロトコル

SLMの理解不足を補うためのツール呼び出し通信レイヤー:

```
1. ツール呼び出し前にスキーマ確認（Locking Context）
2. 曖昧なパラメータの検出
   例: 日付形式（MM/DD vs DD/MM）、通貨単位
3. 曖昧さがあれば実行前にQuery for Clarification
4. 確認後に実行
```

**効果:** ツール呼び出し失敗率を劇的に低下。
**SLMでの重要性:** LLMなら暗黙的に解決する曖昧さを、SLMは見逃す。ハーネスで明示的に捕捉。

## 累進的開示（Progressive Disclosure）

SLMの限られたコンテキストを最大活用するための情報構成:

```
100行の「地図」（ルートファイル）
  ├── 設計判断へのポインタ → 必要時に読み込み
  ├── 実行計画へのポインタ → 必要時に読み込み
  └── 仕様書へのポインタ → 必要時に読み込み
```

- 全情報を一度に渡さない
- エージェントが必要な時に必要な情報へ「航海」できる設計
- コンテキストウィンドウの枯渇を防ぎつつ情報密度を高める

OpenAIの実験で「AGENTS.mdという巨大指示ファイルを廃止し、100行の地図に置き換えた」ことで
エージェントのタスク成功率が向上した事例あり。
