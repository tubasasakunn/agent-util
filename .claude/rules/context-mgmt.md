# internal/context/ ルール

## 責務

コンテキスト（メッセージ履歴）の管理を担当する:
- メッセージの追加・参照・削除
- トークン数の推定と使用率の監視
- 縮約カスケード（バジェット制限 → Snip → Compact）
- コンテキスト分割（サブエージェント用のfork）
- システムプロンプトの静的/動的分離

## メッセージ管理

メッセージ履歴はスライスで管理する。各メッセージはロールとトークン数推定値を持つ。

```go
type Message struct {
    Role      Role            // system, user, assistant, tool
    Content   string
    ToolCalls []ToolCall      // assistant のみ
    ToolID    string          // tool のみ
    Tokens    int             // 推定トークン数
}

type Manager struct {
    messages   []Message
    tokenLimit int
    tokenCount int
}
```

## トークン数推定

- 厳密なtokenizerを使わず、文字数ベースの推定でよい（日本語: 1文字≒1.5トークン、英語: 1単語≒1.3トークン）
- システムプロンプト + ツール定義 + メッセージ履歴の合計で管理する
- 使用率（tokenCount / tokenLimit）を常に計算可能にする
- 閾値超過時にコールバック/イベントを発火する仕組みを持つ

## 縮約カスケード

コンテキストが閾値（例: 80%）を超えたら段階的に縮約する。
各段階は独立した関数として実装し、カスケードの順序は設定で変更可能にする。

```
段階1: BudgetTrim — ツール結果の文字数制限（超過分を切り捨て）
段階2: Snip — 古いメッセージを削除（直近N件は保持）
段階3: Compact — LLMで古いメッセージを要約して置換
```

- 各段階は `func(messages []Message) []Message` のシグネチャで統一する
- 縮約後にトークン数を再計算する
- 直近のメッセージ（最低5件）は縮約対象外とする
- システムメッセージは縮約しない

## コンテキスト分割

サブエージェント用にコンテキストをforkする仕組み。

```go
func (m *Manager) Fork(systemPrompt string) *Manager {
    // 新しいManagerを作成
    // システムプロンプトだけを持つクリーンなコンテキスト
    // 親のメッセージ履歴は含めない
}
```

- forkしたコンテキストは親と完全に独立する
- サブエージェントの結果だけを親に返す（結果の要約として）

## システムプロンプトの構成

```
[静的部分] — セッション間で変わらない
  基本ルール
  ツール一覧（ルーター用テキスト）
  制約・禁止事項
[__DYNAMIC_BOUNDARY__]
[動的部分] — セッション固有
  現在のタスク記述
  環境情報
```

静的部分を前、動的部分を後ろに配置する。

## ルール

- メッセージの追加は必ず Manager のメソッドを通す（直接スライス操作禁止）
- トークン数は追加時に計算してキャッシュする（毎回全計算しない）
- 縮約は副作用を伴う操作なので、呼び出し側が明示的にトリガーする
- Manager はスレッドセーフにする（sync.Mutex で保護）
- コンテキストの状態変更時にイベントを通知する仕組み（Observer パターン）を持つ
